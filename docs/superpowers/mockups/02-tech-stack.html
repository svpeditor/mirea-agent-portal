Рекомендуемый tech-stack ядра

Стабильное ядро портала. Агенты внутри Docker-sandbox могут быть на чём угодно — на стек ядра это не влияет.

Frontend

Next.js 15 + TypeScript + Tailwind + shadcn/ui

React-стек — индустриальный стандарт, востребованный в портфолио. App Router хорошо ложится на «страницы агентов / задач». shadcn/ui даёт production-компоненты без часов вёрстки.
Альтернатива: HTMX + Jinja — быстрее писать, но скучнее студентам
Backend API

FastAPI + Pydantic v2 + SQLAlchemy 2.0 async + asyncpg

Python-нативный, типизированный, async по умолчанию. Идеально стыкуется с Python-агентами. Pydantic-схемы автоматически становятся OpenAPI-доками — бесплатный материал для команды «агенты».
Альтернатива: Django REST Framework — даёт готовую админку, но тяжелее
База

PostgreSQL 16 + Alembic (миграции)

JSONB для гибких параметров агентов, full-text search для названий/описаний. Не обсуждается для всего, что больше тривиального.
Очередь

RQ (Redis Queue) + Redis

Простой и проверенный для нашего масштаба (десятки задач в час). Гораздо проще Celery, на чистом Python. Воркер запускает Docker-контейнер через Docker SDK.
Альтернатива: Postgres-based queue без отдельного Redis
Sandbox

Docker — один контейнер на одну задачу

Лимиты CPU/RAM/времени. Кривой студенческий агент не валит портал. Зависимости агентов не конфликтуют. Образ агента собирается при добавлении в систему.
Auth

email + пароль → JWT

Регистрация по приглашению админа — преподаватели небольшое замкнутое сообщество. Если МИРЭА когда-то даст SSO/OAuth — добавим провайдер.
LLM-gateway

OpenRouter через прокси портала

Один ключ → Claude / GPT / Gemini / DeepSeek. Агент в манифесте указывает модель, портал инжектит токен в env при запуске контейнера. Все вызовы — через прокси, чтобы считать расход и применять лимиты по пользователю/агенту.
Деплой

Docker Compose → один сервер

Все сервисы в compose-файле. git pull && docker compose up -d. На втором этапе — CI через GitHub Actions с push-on-deploy.
Альтернатива: Kubernetes — overkill для нашего масштаба
Хранилище

Локальная FS на сервере → S3-совместимое позже

Загрузки + артефакты в директории, прибитой к Postgres. Когда упрёмся в место — переедем на S3 без изменения API (через адаптер).
