Все таски на одного implementer'а — много файлов одного агента, не параллелится по делу.

#### T1.1 — manifest.yaml + boilerplate

**Файлы:** `agents/proverka_stub/manifest.yaml`, `agents/proverka_stub/requirements.txt`, `agents/proverka_stub/Dockerfile`, `agents/proverka_stub/README.md`

**DoD:**
- `manifest.yaml` строго по спеке (см. `2026-05-10-1.4-agents-port-design.md`).
- `requirements.txt`: `python-docx`.
- `Dockerfile` — копия из `agents/echo/Dockerfile` с поправленным комментарием (для local-dev, портал генерирует свой).
- `README.md`: «как запустить через `portal-sdk-run-local`» + «как опубликовать через `/api/admin/agents`» + блок «Real-версия живёт в …, в этом stub'е mock-оценка».

#### T1.2 — agent.py

**Файлы:** `agents/proverka_stub/agent.py`

**Входные:** `params={"competition": "ВНТК-2025", "grade_level": "10-11"}`, папка `$INPUT_DIR/works/<folder1>/...`, `<folder2>/...`, ...

**Выход:** `$OUTPUT_DIR/report.docx` (Word-таблица), `$OUTPUT_DIR/per_work.zip` (zip с per-work-заглушками).

**DoD:**
- Использует `portal_sdk.Agent`.
- Сканирует `agent.input_dir("works")` — итерируется по подпапкам (Path.iterdir + is_dir).
- На каждой подпапке: progress + item_done.
- `report.docx` через `python-docx`: heading + таблица с колонками «Работа», «Файлов», «Размер», «Mock-оценка».
- `per_work.zip` собирается через `zipfile.ZipFile` — на каждую работу один `<folder_name>.docx` с одной строкой текста.
- Финал — `agent.result(artifacts=[...])`.
- Файл проходит `ruff check` (на этих файлах).

#### T1.3 — manifest validation тест

**Файлы:** `agents/proverka_stub/tests/test_manifest.py`

**DoD:**
- Импортирует Pydantic-схему из `portal_sdk.manifest`.
- Парсит `manifest.yaml` → проверяет, что валидируется без ошибок.
- `pytest agents/proverka_stub/tests/` зелёный.
