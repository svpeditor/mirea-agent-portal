## 1. Цель и контекст

Поднять минимальный backend, на который потом сядут реестр агентов, очередь задач и frontend. На этом этапе видимый результат для оператора — возможность через `curl` зарегистрироваться по invite-ссылке и получить JWT-сессию.

**Что МОЖНО сделать после merge 1.2.1:**

- Админ создаёт invite-ссылку → отдаёт студенту → студент регистрируется → логинится → получает `me`.
- Параллельно может стартовать План 1.2.2 (реестр), потому что admin-роль и auth-инфраструктура готовы.
- Параллельно может стартовать План 1.3 (frontend) — есть стабильный API для страниц login / register / me.

## 2. Scope

### Входит ✅

- FastAPI 0.115+ с async-стеком (Pydantic v2, SQLAlchemy 2.0 async, asyncpg).
- PostgreSQL 16 + Alembic миграции.
- Регистрация по одноразовым invite-токенам (e-mail + password + display_name).
- Login / logout / refresh с JWT в HttpOnly cookies.
- Refresh-token rotation с reuse-detection.
- Базовые роли `user` / `admin`.
- Bootstrap первого админа из ENV при пустой БД.
- Admin-эндпоинты для управления юзерами (CRUD, reset-password) и invite-ссылками.
- `docker-compose.yml` с Postgres + API, hot-reload в dev.
- pytest-тесты на реальном Postgres через `testcontainers`, ≥85% coverage.

### Сознательно отложено ⏭

- Реестр агентов и всё с ним связанное (Планы 1.2.2-1.2.4).
- Frontend (1.3).
- TLS / публичный домен / наружный доступ (отдельная задача после 1.2.3, либо когда у Дани появится белый IP / VPS-релей).
- Email-нотификации, SMTP-инфраструктура.
- Magic-link «forgot password» (вместо этого админ ручно сбрасывает).
- 2FA, OAuth, SSO МИРЭА.
- CI/CD GitHub Actions (Спек 5).
- Rate-limiting на эндпоинты (закрытый портал по invite, флуда не будет).
- Audit-log в отдельной таблице (хватает `refresh_tokens.user_agent` + structlog).
- Soft-delete юзеров.

### Сознательно НЕ делаем 🚫

- Регистрация без invite-токена.
- Самовосстановление пароля юзером (только через админа).
- Хранение секретов в чём-либо помимо `.env` файла (Vault / 1Password — overkill для MVP).

## 4. Модель данных

### `users`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | `gen_random_uuid()` |
| `email` | text UNIQUE NOT NULL | lower-cased на входе через Pydantic-валидатор |
| `password_hash` | text NOT NULL | bcrypt cost=12 (на тестах cost=4 через monkeypatch) |
| `display_name` | text NOT NULL | то, что ввёл при регистрации |
| `role` | text NOT NULL CHECK (`role IN ('user','admin')`) | default `'user'` |
| `monthly_budget_usd` | numeric(10,2) NOT NULL | default `5.00` (квоты 1.2.4 — поле создаём заранее, чтобы не делать миграцию) |
| `created_at` | timestamptz NOT NULL | default `now()` |
| `updated_at` | timestamptz NOT NULL | trigger на UPDATE |

### `invites`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | |
| `token` | text UNIQUE NOT NULL | секретный, 32 случайных байта в base64url; токен НЕ хешируется — это разовая ссылка с коротким TTL, не пароль |
| `email` | text NOT NULL | для какого email — на регистрации проверяем совпадение |
| `created_by_user_id` | UUID FK users(id) ON DELETE RESTRICT NOT NULL | какой админ создал. RESTRICT, чтобы не терялась история — для удаления админа сначала надо удалить/перевесить его invites |
| `expires_at` | timestamptz NOT NULL | created_at + 7 days |
| `used_at` | timestamptz NULL | NULL — токен живой; не NULL — уже использован/отменён |
| `used_by_user_id` | UUID FK users(id) ON DELETE SET NULL | для аудита (или какой админ отменил). SET NULL, чтобы удаление юзера не ломало запись invite |
| `created_at` | timestamptz NOT NULL | |

### `refresh_tokens`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | |
| `user_id` | UUID FK users(id) ON DELETE CASCADE NOT NULL | |
| `token_hash` | text NOT NULL | SHA-256 от сырого refresh-токена, не сам токен |
| `expires_at` | timestamptz NOT NULL | created_at + 30 days |
| `revoked_at` | timestamptz NULL | для логаута / ротации |
| `replaced_by_id` | UUID FK refresh_tokens(id) ON DELETE SET NULL | при ротации указывает на новый токен (для обнаружения reuse-attack) |
| `user_agent` | text NULL | для админа: «откуда логинились» |
| `ip` | inet NULL | для админа |
| `created_at` | timestamptz NOT NULL | |

### Индексы

- `users.email` — UNIQUE (создан constraint'ом).
- `invites.token` — UNIQUE.
- `invites(email) WHERE used_at IS NULL` — частичный, для проверки «есть ли живой инвайт на этот email».
- `refresh_tokens(user_id) WHERE revoked_at IS NULL` — частичный, для активных токенов юзера.

### Refresh-token rotation: алгоритм

1. Login: генерим access (24ч JWT, не хранится в БД) + refresh (30д, хеш в БД). Оба — в HttpOnly cookies.
2. Через 24ч клиент шлёт `POST /api/auth/refresh` с обоими cookies (access уже просрочен — это нормально, refresh жив).
3. API: `sha256(refresh_from_cookie)` → ищет в `refresh_tokens` → проверяет `revoked_at IS NULL` и `expires_at > now()`. Если ок:
   - создаёт новый access + новый refresh
   - старый refresh: `UPDATE refresh_tokens SET revoked_at = now(), replaced_by_id = <новый.id>`
   - в response: новые cookies
4. **Reuse detection**: если предъявлен refresh с `revoked_at IS NOT NULL` — значит украли (пользователь уже его обновил, копия из лог-файла / cookie-кражи). Действие:
   - `UPDATE refresh_tokens SET revoked_at = now() WHERE user_id = <usr> AND revoked_at IS NULL` (revoke ВСЕ активные refresh этого юзера)
   - response: `401 REFRESH_REUSE_DETECTED`, чистим cookies
   - в лог: `WARNING` с `user_id`, `user_agent` обеих сессий, `ip` обеих

## 5. API endpoints

Все ответы — JSON, ошибки — `{"error": {"code": "...", "message": "..."}}`. Базовый префикс `/api`.

### Публичные

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/health` | `{"status":"ok"}` для оркестратора |
| `POST` | `/api/auth/register` | По invite-токену создать юзера. Вход: `{token, email, display_name, password}`. Проверки: токен живой + не использован + email совпадает с инвайтом. Атомарно: создаёт user, помечает invite `used_at + used_by_user_id`. Возвращает 201 + access+refresh cookies. |
| `POST` | `/api/auth/login` | Вход: `{email, password}`. При успехе: cookies + 200 + `{user: {...}}`. При ошибке — 401 `INVALID_CREDENTIALS` (одинаковый текст для «нет юзера» и «неверный пароль» — без user enumeration). |
| `POST` | `/api/auth/refresh` | Берёт refresh из cookie, ротирует. Ставит новые cookies. На reuse-attack — revoke all + 401. |
| `POST` | `/api/auth/logout` | Берёт refresh из cookie, помечает `revoked_at=now()`. Чистит cookies. |

### Залогинены (`Depends(get_current_user)`)

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/me` | `{id, email, display_name, role, monthly_budget_usd, created_at}` |
| `PATCH` | `/api/me` | Изменить `display_name`. Email — нельзя (ломает аудит). |
| `POST` | `/api/me/change-password` | `{current_password, new_password}`. Проверяет current, хеширует new. **Отзывает все refresh-токены кроме текущего.** |

### Только админ (`Depends(require_admin)`)

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/admin/users` | Список юзеров. Пагинация `?limit=50&cursor=<id>`. |
| `GET` | `/api/admin/users/{id}` | Один юзер. |
| `PATCH` | `/api/admin/users/{id}` | Изменить `role`, `monthly_budget_usd`, `display_name`. Email не трогаем. |
| `DELETE` | `/api/admin/users/{id}` | Хард-удаление. Cascade `refresh_tokens`. Invites не трогаем (FK NULL on delete). |
| `POST` | `/api/admin/users/{id}/reset-password` | Сгенерить случайный временный пароль (16 символов, base64url), сохранить хеш, **revoke все refresh** этого юзера. Возвращает админу `{temporary_password}` в response. Админ передаёт лично. |
| `POST` | `/api/admin/invites` | `{email}`. Создать invite-токен. **Проверки:** (а) если email уже зарегистрирован как юзер — 409 `EMAIL_ALREADY_REGISTERED`; (б) если на email есть живой непрошедший invite — 409 `INVITE_ALREADY_PENDING` с указанием существующего id. Возвращает: `{token, expires_at, registration_url}` где url — `<base>/register?token=...` (фронт подхватит позже, на этом этапе это удобство для админа). |
| `GET` | `/api/admin/invites` | Список инвайтов с фильтром `?status=active|used|expired`. |
| `DELETE` | `/api/admin/invites/{id}` | Отменить инвайт (поставить `used_at=now()` и в `used_by_user_id` записать админа). |

### Cookies — параметры

- `access_token`: HttpOnly, Secure (только если `COOKIE_SECURE=true`), SameSite=Strict, Path=`/api`, Max-Age=86400 (24ч).
- `refresh_token`: HttpOnly, Secure, SameSite=Strict, Path=`/api/auth`, Max-Age=2592000 (30д). Path намеренно узкий — refresh не отправляется со всеми запросами.

### Origin check

Middleware проверяет `Origin` (или `Referer` если Origin отсутствует, например в legacy-навигации) для всех `POST/PATCH/DELETE`. Должен совпадать с одним из `ALLOWED_ORIGINS` из конфига. Иначе 403 `ORIGIN_MISMATCH`. На `GET` не проверяем — они идемпотентны.

### Bootstrap админа

При старте API:
1. `SELECT count(*) FROM users`.
2. Если 0 и `INITIAL_ADMIN_EMAIL` + `INITIAL_ADMIN_PASSWORD` заданы → создаём админа.
3. Если 0 и ENV пусты → API падает с `RuntimeError("Пустая БД, но INITIAL_ADMIN_* не заданы — портал не сможет управляться")`.
4. Если в БД уже есть юзеры → ENV игнорируется (никаких автоматических обновлений).

### Безопасность секретов

- `JWT_SECRET`: 64+ случайных байт, `openssl rand -base64 64` → в `.env`. При потере все юзеры разлогинятся.
- `INITIAL_ADMIN_PASSWORD` сменить через `/api/me/change-password` сразу после первого логина. После этого переменную можно удалить из `.env` (читается только при пустой БД).

### Ключевые сценарии

**`test_auth_register.py`:**
- ✓ Валидный invite → юзер создаётся, invite помечен used, cookies стоят
- ✓ Несуществующий token → 400
- ✓ Использованный token → 400
- ✓ Просроченный token (freezegun: создан 8 дней назад) → 400
- ✓ email в теле не совпадает с email в инвайте → 400
- ✓ Пароль короче 8 символов → 422
- ✓ display_name пустой → 422
- ✓ Дважды зарегаться по тому же токену (race condition через `INSERT ... WHERE used_at IS NULL`) → второй получает 400
- ✓ Email с разным регистром → нормализуется в lowercase

**`test_auth_refresh.py`:**
- ✓ Refresh ротируется
- ✓ **Reuse-attack**: использовать уже отозванный refresh → revoke ВСЕ активные refresh + 401 `REFRESH_REUSE_DETECTED`
- ✓ Просроченный refresh → 401, cookies чистятся
- ✓ Logout одной сессии не разлогинивает другие

**`test_security.py`:**
- ✓ Cookies имеют HttpOnly + SameSite=Strict + правильный Path
- ✓ POST с чужим Origin → 403 `ORIGIN_MISMATCH`
- ✓ Login с неверным паролем и login с несуществующим email — одинаковая ошибка (no user enumeration)
- ✓ Bcrypt-хеш реально хеширует (детерминизм отсутствует)
- ✓ `password_hash` никогда не попадает в response

**`test_bootstrap.py`:**
- ✓ Пустая БД + ENV есть → создаётся админ при старте
- ✓ Пустая БД + ENV нет → стартап падает с ясной ошибкой
- ✓ В БД уже есть юзеры + ENV есть → ENV игнорируется

### Bcrypt cost для тестов

В `tests/conftest.py` через monkeypatch на уровне session: `cost=4` вместо `cost=12`. Без этого 60 тестов с bcrypt на cost=12 займут минуты.

## 8. Error handling

### Единый формат

Все 4xx/5xx ошибки →

```json
{"error": {"code": "INVALID_CREDENTIALS", "message": "Неверный email или пароль."}}
```

Для `VALIDATION_ERROR` дополнительно:

```json
{"error": {"code": "VALIDATION_ERROR", "message": "...", "details": [...]}}
```

### Коды

| HTTP | code | Когда |
|---|---|---|
| 400 | `INVITE_INVALID` | invite не существует / уже использован / просрочен / email не совпал |
| 401 | `INVALID_CREDENTIALS` | login: неверный email или пароль |
| 401 | `NOT_AUTHENTICATED` | нет access-cookie или JWT невалидный |
| 401 | `REFRESH_INVALID` | refresh не найден / просрочен / revoked |
| 401 | `REFRESH_REUSE_DETECTED` | предъявлен уже использованный refresh — все revoked |
| 403 | `FORBIDDEN` | юзер не админ для admin-эндпоинтов |
| 403 | `ORIGIN_MISMATCH` | Origin/Referer не в ALLOWED_ORIGINS |
| 404 | `USER_NOT_FOUND` | админ запросил несуществующего |
| 409 | `EMAIL_ALREADY_EXISTS` | регистрация по invite, но email уже занят |
| 409 | `EMAIL_ALREADY_REGISTERED` | админ создаёт invite на уже-зарегистрированный email |
| 409 | `INVITE_ALREADY_PENDING` | админ создаёт invite на email, на который уже есть живой |
| 422 | `VALIDATION_ERROR` | Pydantic-валидация — `details` массивом ошибок полей |
| 500 | `INTERNAL_ERROR` | необработанное — без stack trace в response, с trace в логах |

### Логи

- Структурированный JSON через `structlog`.
- На каждый запрос: `request_id`, `method`, `path`, `status`, `duration_ms`, `user_id`.
- Чувствительные поля (`password`, `current_password`, `new_password`, `token`, `password_hash`) — middleware фильтрует.
- На `REFRESH_REUSE_DETECTED` — `WARNING` с обоими `user_agent` (старый + новый) и `ip` обоих: возможный угон сессии.

## 11. Риски

| Риск | Mitigation |
|---|---|
| `testcontainers` не работает на Mac Mini ARM64 | postgres:16 — multi-arch image, проверим в Task 1 плана реализации |
| asyncpg + SQLAlchemy 2.0 версии конфликтуют | пинануть точные версии в `pyproject.toml`, без upper-bound на minor |
| bcrypt медленный на тестах | в `conftest.py` cost=4 через monkeypatch на уровне session |
| Forgot-password leak через timing attack | `bcrypt.checkpw` constant-time |
| Race condition в register (два запроса с одним token) | `UPDATE invites SET used_at=now() WHERE id=? AND used_at IS NULL RETURNING *` — атомарно |
| `JWT_SECRET` в env-файле может утечь | при ротации все юзеры разлогинятся — это ОК на закрытом портале с ~10 юзерами |
