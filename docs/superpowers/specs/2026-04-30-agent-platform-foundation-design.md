## 1. Цель и контекст

### Что строим

Веб-портал, через который преподаватели кафедры запускают AI-агентов на своих задачах **без знания командной строки и без установки Python**. Студенты-разработчики НУГ пишут новых агентов под опубликованный контракт — портал автоматически их подхватывает.

### Откуда задача

У заказчика (Дани) уже есть рабочие AI-агенты на Python:

- **`science_agent`** — анализирует папку с PDF научных статей по введённой теме, выдаёт Word-отчёт с ранжированием по релевантности и обзором литературы. Работает на DeepSeek API.
- **`proverka`** — анализирует папку с конкурсными работами школьников (по подпапке на работу: текст + презентация), выдаёт Word с экспертными заключениями и сводной таблицей вердиктов. Реальный масштаб — 46 работ за прогон.

Оба запускаются `python main.py` через `input()`, имеют DOCX-инструкцию для конечных пользователей с Win+R и установкой Python. Преподаватели по этой инструкции работать не могут или не хотят.

### Почему отдельный портал, а не «улучшим скрипты»

- Задача не одна — набор агентов будет расти. Унификация интерфейса нужна с самого начала.
- Несколько групп студентов работают параллельно: «портал» и «агенты» — разные команды. Без чёткого контракта между ними любая правка превращается в координационный кошмар.
- Преподаватели — не программисты. Им нужен браузерный UI, прогресс-бар, скачивание результата, история запусков, повтор с теми же параметрами.
- Безопасность: API-ключи сейчас захардкожены в `config.py`. На портале они должны быть централизованы, иначе их у нас будет N штук в разных репозиториях, и любой утёк = пересоздавать всё.

### Кто пользуется

| Роль | Сценарий |
|---|---|
| **Преподаватель** | Логинится, выбирает агента из плиток, загружает свои файлы, ждёт прогресс, скачивает Word. |
| **Студент-разработчик НУГ (агенты)** | Пишет агента у себя локально по контракту, пушит в свой Git, добавляет URL в админку портала. |
| **Студент-разработчик НУГ (портал)** | Поддерживает само ядро портала, контракт, документацию. |
| **Админ** | Управляет вкладками, агентами, юзерами, лимитами. На старте — Даня. |

## 2. Scope этого спека

### Входит в MVP ✅

1. **Контракт агента** — `manifest.yaml`, runtime-протокол I/O, Python-SDK как обёртка
2. **Echo-агент** — reference-имплементация, тривиальная, как пример для студентов
3. **Перенос `proverka` под контракт** — боевой тест контракта на реальном агенте
4. **Портал**: главная (вкладки + плитки), страница агента (форма), страница задачи (live-прогресс), админка (вкладки/агенты/юзеры/задачи/лимиты)
5. **Auth**: email + пароль, регистрация по приглашению админа
6. **Job queue** + Docker-sandbox per job + жёсткие лимиты CPU/RAM/времени
7. **OpenRouter-прокси** — централизованный gateway к LLM с учётом расхода
8. **Квоты** — на пользователя, на портал, whitelist моделей
9. **Деплой** — Docker Compose на Mac Mini M4 (dev = staging = prod на старте)

### Сознательно отложено в следующие спеки ⏭

- **Спек 2** — перенос `science_agent` под контракт + полировка контракта по найденным несостыковкам
- **Спек 3** — новый агент «Поиск научных статей» (arXiv + Semantic Scholar + Crossref). Новый паттерн контракта: вход = только текст, выход = ranked list. Возможные баги контракта вылезут именно здесь
- **Спек 4** — полноценный визуальный конструктор агентов через UI для не-программистов (drag-n-drop пайплайнов). В MVP остаётся только вкладка «С нуля» с шаблонной формой манифеста
- **Спек 5** — документация для писателей агентов и SDK на других языках (Node.js, Go) — параллельно со Спеком 2

### Сознательно НЕ делаем 🚫

- SSO МИРЭА / OAuth — пока МИРЭА не даст доступ
- 152-ФЗ-комплаенс / переезд на Yandex Cloud — рассмотрим при росте, на старте закрытый портал кафедры
- S3-совместимое хранилище — пока локальная FS, переедем при первой проблеме
- CI/CD GitHub Actions — пока ручной `git pull && docker compose up -d`
- Мультиязычный UI — только русский
- Зависимости между полями формы (`показать X если Y=true`) — отложено
- Полноценный audit log — на старте простой текстовый лог в БД

## 4. Tech-stack

| Слой | Выбор | Причина |
|---|---|---|
| Frontend | Next.js 15 + TypeScript + Tailwind + shadcn/ui | React в портфолио студентов, App Router, production-компоненты |
| Backend | FastAPI + Pydantic v2 + SQLAlchemy 2.0 async + asyncpg | Python-нативность стыкует с агентами, типизация ловит баги до runtime, OpenAPI бесплатно |
| База | PostgreSQL 16 + Alembic | JSONB для гибких параметров, full-text search, advisory locks |
| Очередь | RQ + Redis | Проще Celery, на нашем масштабе (десятки задач/час) хватает |
| Sandbox | Docker per job (через Docker SDK для Python) | Лимиты, изоляция зависимостей, кривой агент не валит портал |
| Auth | email + пароль → JWT (24ч access, 30 дней refresh) | Простой старт; хук под OAuth-провайдеры заложен в схеме |
| LLM gateway | OpenRouter через прокси портала | Один ключ → любая модель; портал считает расход и применяет лимиты |
| Хранилище | Локальная FS (`./storage/...`), привязка через PG | На старте проще; адаптер к S3 в будущем — без изменения API |
| Деплой | Docker Compose на Mac Mini M4 | Один файл, `git pull && docker compose up -d` |

## 5. Контракт агента

### 5.1 `manifest.yaml`

Лежит в корне репозитория агента. Портал читает при добавлении.

```yaml
# Идентификация
id: school-works-expert        # уникальный slug
name: "Эксперт конкурсных работ школьников"
version: "1.0.0"               # семвер; рекомендуется тэгом git
category: "научная-работа"     # одна из вкладок портала
icon: "🎓"
short_description: |
  Анализирует работы школьников по чек-листу и выдаёт
  экспертное заключение в Word.
about: |
  # Длинное описание (markdown)
  Принимает папку с подпапками-работами, в каждой из которых
  должны лежать текст работы и презентация. ...

# Параметры формы — портал автоматически рисует UI
inputs:
  section:
    type: text                 # text|textarea|number|checkbox|select|radio|date
    label: "Секция конкурса"
    placeholder: "например: робототехника"
    default: "общая"
    required: true
  strict_mode:
    type: checkbox
    label: "Жёсткий режим"
    help: "Снижает балл за каждый недостаток"
    default: false

# Файлы на входе
files:
  works_folder:
    type: folder               # single_file|multi_files|folder|zip
    label: "Папка с работами"
    help: "В каждой подпапке — текст работы и презентация"
    accept: [".docx", ".pdf", ".pptx"]
    max_total_size_mb: 500
    required: true

# Что возвращает агент
outputs:
  - id: report
    type: docx                 # docx|pdf|xlsx|zip|html|json|any
    label: "Экспертное заключение"
    filename: "экспертное_заключение.docx"
    primary: true              # большая кнопка скачать на странице задачи
  - id: summary
    type: json
    label: "Структурированный результат"
    filename: "summary.json"

# Runtime
runtime:
  docker:
    base_image: "python:3.12-slim"
    setup:
      - "pip install -r requirements.txt"
    entrypoint: ["python", "agent.py"]

  llm:
    provider: openrouter
    models: ["deepseek/deepseek-chat"]   # whitelist моделей агента

  limits:
    max_runtime_minutes: 60
    max_memory_mb: 1024
    max_cpu_cores: 2
```

### 5.2 Runtime I/O протокол

Между процессом-агентом внутри Docker и воркером портала.

**Что портал даёт агенту:**

| Канал | Содержимое |
|---|---|
| `$PARAMS_FILE` (env) | Путь к JSON с параметрами формы. Агент читает один раз. |
| `$INPUT_DIR` (env) | Путь к bind-mount директории. Внутри — подкаталог на каждое `files.<id>` из манифеста. Структура подпапок исходных файлов сохраняется. |
| `$OUTPUT_DIR` (env) | Путь к пустой bind-mount директории, куда агент кладёт артефакты. |
| `$OPENROUTER_API_KEY`, `$OPENROUTER_BASE_URL` (env) | Прокси портала, через который агент дёргает LLM обычным OpenAI-клиентом. |

**Что агент пишет порталу:**

- `stdout` — построчный NDJSON. Каждая строка — JSON-событие. Парсится без буферизации.
- `stderr` — свободный текст. Складывается в логи задачи как есть. Сюда же `print()`, traceback, отладка.
- Файлы — в `$OUTPUT_DIR/` с именами, совпадающими с `outputs[].filename` из манифеста.

**События в stdout:**

| Тип | Поля | Когда |
|---|---|---|
| `started` | `ts` | Агент стартанул. Портал переводит job в «выполняется». |
| `progress` | `value` (0.0–1.0), `label` | Числовой прогресс + подпись. UI рисует прогресс-бар. |
| `log` | `level` (`info`/`warn`/`error`/`debug`), `msg` | Сообщение пользователю в общую ленту. |
| `item_done` | `id`, `summary`, `data?` | Завершение одного элемента в серии. UI показывает «✓ <id> — <summary>». |
| `error` | `id?`, `msg`, `retryable?` | Нефатальная ошибка. Агент продолжает. UI помечает элемент красным. |
| `result` | `artifacts: [{id, path}]` | Успешное завершение. Портал индексирует артефакты, открывает страницу скачивания. |
| `failed` | `msg`, `details?` | Критическое падение. Job → failed. Артефакты, если есть, доступны для дебага. |

**Пример полного стрима:**

```ndjson
{"type":"started","ts":"2026-04-30T15:01:00Z"}
{"type":"progress","value":0.0,"label":"Обработка 01"}
{"type":"log","level":"info","msg":"найдены файлы: текст.docx, презентация.pptx"}
{"type":"item_done","id":"01","summary":"НЕ РЕКОМЕНДОВАНА","data":{"verdict":"reject","scientific":5.0}}
{"type":"progress","value":0.022}
{"type":"item_done","id":"02","summary":"РЕКОМЕНДОВАНА","data":{"verdict":"accept","scientific":8.0}}
{"type":"progress","value":1.0}
{"type":"result","artifacts":[{"id":"report","path":"экспертное_заключение.docx"},{"id":"summary","path":"summary.json"}]}
```

### 5.3 Python-SDK

Студент не пишет JSON руками, использует `portal_sdk`:

```python
from portal_sdk import Agent
from openai import OpenAI
from docx import Document

agent = Agent()
params = agent.params                          # dict, читается из $PARAMS_FILE
section = params["section"]
works_dir = agent.input_dir("works_folder")    # pathlib.Path к bind-mount
output_dir = agent.output_dir                  # pathlib.Path, куда писать артефакты

llm = OpenAI(
    api_key=agent.env["OPENROUTER_API_KEY"],
    base_url=agent.env["OPENROUTER_BASE_URL"],
)

results = []
folders = sorted(works_dir.iterdir())
for i, folder in enumerate(folders):
    agent.progress(i / len(folders), f"Обработка {folder.name}")
    try:
        result = analyze_work(llm, folder, section)
        results.append(result)
        agent.item_done(folder.name, summary=result["verdict"], data=result)
    except Exception as e:
        agent.error(folder.name, str(e))

# Студент сам пишет файлы; SDK НЕ навязывает форматов
doc = Document()
build_report(doc, results)
doc.save(output_dir / "экспертное_заключение.docx")

(output_dir / "summary.json").write_text(json.dumps(results, ensure_ascii=False))

agent.result(artifacts=["report", "summary"])  # id из манифеста, не имена файлов
```

SDK живёт в репозитории портала, публикуется в Docker base image — установка `pip install portal_sdk` уже в нём предустановлена.

### 5.4 Доставка агента

Три способа добавления в админке:

1. **Git URL (основной)** — портал клонирует репо, читает manifest, собирает Docker-образ. Версия — git tag.
2. **ZIP-загрузка** — для не-программистов и быстрых проб. Аплоад через UI, всё остальное так же.
3. **С нуля** — встроенная форма заполнения манифеста + редактор `agent.py` в браузере. Минимальный конструктор для MVP, не drag-n-drop.

После добавления:
1. Парсинг и валидация `manifest.yaml`.
2. Сборка Docker-образа (асинхронно, статус виден в админке).
3. Если сборка успешна — агент появляется в плитках своей вкладки.
4. Если упала — failed-агент видим только админу с причиной.

## 6. Модель данных

```

               cost_usd, created_at, started_at, finished_at)

job_files     (id, job_id, kind ['input'|'output'], path, size_bytes, mime)

```

Файлы — в FS по schema:
```

  jobs/<job_id>/
    input/<input_id>/<original_structure>
    output/<artifact_filename>
```

## 7. UI

Все мокапы в `docs/superpowers/mockups/` (в репозитории, не в `.superpowers/`).

- **Главная** — сайдбар с категориями слева (3 вкладки + «мои задачи» + «админка» снизу для админа), плитки агентов справа в 2 колонки. Выбран на брейнштормe (вариант A).

- **Страница задачи (done)** — большая кнопка скачать primary-артефакт + список остальных + сводка (автоматически из `item_done.data` event'ов) + «Запустить ещё раз с теми же параметрами».
- **Админка** — табы вверху: вкладки / агенты / пользователи / задачи / лимиты / логи. На странице агентов — список с toggle on/off и боковая панель добавления.

## 8. Лимиты и квоты

Четыре уровня защиты:

| Уровень | Что | Где задаётся | Действие при превышении |
|---|---|---|---|
| Агент | `max_runtime_minutes`, `max_memory_mb`, `max_cpu_cores` | `manifest.yaml` | Контейнер убивается, job → failed |
| Запуск | Оценка стоимости перед стартом | Автоматически из объёма входа × модель | Подтверждение если > 50% оставшегося бюджета юзера |
| Пользователь | Месячный бюджет в USD (по умолчанию **$5**) | Админка | Новые задачи отклоняются до начала месяца, текущие догоняются |
| Портал | Глобальный месячный потолок **$50** | Конфиг + UI админа | Kill-switch: всем новым задачам отказ |

**Whitelist моделей.** По умолчанию доступен только `deepseek/deepseek-chat` (~$0.14/M токенов). `claude-sonnet-4`, `gpt-4` — только если агент указал в манифесте и админ апрувнул при добавлении агента.

**Дашборд расхода** в админке — top-10 пользователей и top-10 агентов по тратам, чтобы быстро находить «прожорливых».

## 9. Безопасность

- API-ключ OpenRouter инжектится в env контейнера агента, не лежит в коде агента, не доступен вне контейнера.
- Все исходящие LLM-запросы — через прокси портала (URL `OPENROUTER_BASE_URL` указывает на эндпоинт портала). Это даёт учёт расхода, лимиты, аудит.
- Контейнер запускается с `--network none` для всего, кроме прокси портала. Студенческий агент не может ходить в произвольный интернет.
- Контейнер с `--read-only` корневой FS (кроме `/tmp` и `$OUTPUT_DIR`).
- Пользовательские пароли — bcrypt.
- JWT в httpOnly cookie. Refresh token rotation.
- Загрузка файлов: проверка расширения по `accept`, проверка реального типа через `python-magic`, лимит размера, антивирус ClamAV для файлов > 1 МБ (опционально, если кафедре нужно).
- 152-ФЗ — отложен. На старте — закрытый портал кафедры с регистрацией по приглашению.

```

  redis:
  api:           # FastAPI
  worker:        # RQ-воркер, имеет доступ к /var/run/docker.sock
  frontend:      # Next.js
  caddy:         # reverse proxy + автоTLS
```

Доменное имя — поддомен на твоей зоне (например, `portal.<домен>.ru`). На входе — Caddy с автоматическим Let's Encrypt.

Cloudflare Tunnel НЕ используем сразу (по твоей памяти `*.dwkch.ru → 530` — тоннели у нас уже умирали, ТСПУ режет CF edge). Если домашняя сеть будет подло тормозить — переключимся на Hetzner CX32 (это эскалация, не часть MVP).

`git pull && docker compose up -d --build`. CI/CD — отложен в Спек 5.

## 11. Критерии успеха MVP

Спек считается выполненным, когда:

1. ✅ Преподаватель открывает `https://portal.<домен>` в Safari/Chrome, логинится по выданным админом credentials, выбирает «Эксперт работ школьников», загружает свою папку из 35-46 работ, видит прогресс-бар обновляющийся в реальном времени, через ~30 минут скачивает Word и открывает его — это тот же `экспертное_заключение.docx`, что давал ему `proverka` локально (или лучше).
2. ✅ Студент НУГ берёт свежий клон репозитория `agent-template`, меняет 2-3 строки в `agent.py`, пушит в свой Git, добавляет URL в админке портала — через 2 минуты его агент появляется в плитках.
3. ✅ Админ может: добавить новую вкладку, переместить агента из вкладки в вкладку, выключить агента одним кликом, увидеть текущий расход за месяц, поднять кому-то бюджет.
4. ✅ Любой агент с бесконечным циклом убивается через `max_runtime_minutes`, ничего не валит вокруг, job помечается failed с причиной.
5. ✅ Превышение месячного бюджета пользователя или портала — kill-switch срабатывает, новые задачи отклоняются с понятным сообщением.

## 13. Риски и открытые вопросы

| Риск / вопрос | Импакт | План |
|---|---|---|
| Mac Mini как прод — нестабильный сетевой канал | Средний | Падения мониторим, при первом устойчивом инциденте — Hetzner CX32 |
| 152-ФЗ vs ФИО школьников в proverka | Средний (на закрытом портале — низкий) | Закрытая регистрация, anti-PII checklist при загрузке. При выходе наружу — Yandex Cloud + ОПД |
| OpenRouter-биллинг — суммы могут расти быстрее ожидаемого | Низкий–средний | Глобальный $50 потолок + дашборд + отдельный алерт админу при 80% бюджета |
| Студенческий агент с уязвимостью (RCE через зависимость) | Низкий за счёт sandbox | `--network none` + read-only FS + non-root user в контейнере |
| Большие файлы (500 МБ папок proverka) — IO-боттлнек | Средний | Stream-upload через resumable.js, антивирус только > 1 МБ, мониторинг IO |
| Команда агентов отстаёт от команды портала | Высокий | После echo-агента (неделя 2) команды могут работать параллельно. Раз в неделю синк по контракту. |
| Контракт не покрывает что-то в реальном агенте | Высокий | Именно поэтому в Спек 1 включён перенос proverka. Если что-то не покрывает — дополняем контракт прямо в Спеке 1, не в Спеке 2. |
| Обновление агента (v1 → v2) — не сломаем ли запущенные задачи | Низкий | Каждая задача привязана к конкретному `agent_version_id`. Старая версия живёт, пока есть незавершённые задачи. |
