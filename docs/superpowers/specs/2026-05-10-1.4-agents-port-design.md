# Спека 1.4 — Перенос реальных агентов под контракт v0.1

**Дата:** 2026-05-10
**Статус:** stub-first
**Автор:** autonomous session

## Проблема

Платформа технически готова (1.2.x закрыты, 1.3 frontend в PR #6), но в каталоге сейчас один echo-агент. Без полезных агентов портал не запустить в НУГ. Нужно перенести существующие Python-агенты Дани под контракт SDK v0.1:

- `proverka` — анализ конкурсных работ школьников (≈46 работ за прогон, чек-лист научной экспертизы → Word-заключения)
- `science_agent` — поиск научных статей (DeepSeek-R1 + arXiv/SemScholar/Crossref → Word-отчёт с ранжированием)

## Ограничение текущей сессии

Без явного `OK` от Дани реальные репозитории клонировать нельзя — там может быть proprietary код, ключи, специфические зависимости. Поэтому сессия закрывает stub-уровень: доказать, что обвязка работает на агентах с тем же shape (file-input → docx-output), как у real-версий, чтобы потом было нечего переписывать.

Real-версии сделают в milestone-1 после ОК Дани.

## Цель stub-уровня (milestone-0)

1. В каталоге портала появляются `proverka` и `science_agent` (publish-able через `/api/admin/agents`).
2. Шаблон manifest для каждого совпадает с тем, что нужен real-версиям. Когда real-код приедет, поменяется только `agent.py`, `requirements.txt`, `runtime.setup`.
3. Stub-агенты подсвечивают граничные кейсы, которые echo не покрывает:
   - **proverka**: input `type: folder` с реальной структурой подпапок (одна работа = одна папка с PDF/DOCX).
   - **science_agent**: input `type: text` (тема), интеграция с LLM-прокси через `runtime.llm` (план 1.2.4).

## Целевые manifest'ы (общие для stub и real)

### proverka

```yaml
id: proverka
name: "Проверка конкурсных работ"
version: "0.1.0"
category: "научная-работа"
icon: "📋"
short_description: |
  Принимает папку с работами школьников (PDF/DOCX),
  формирует Word-заключение с оценкой по чек-листу научной экспертизы.

inputs:
  competition:
    type: text
    label: "Название конкурса"
    placeholder: "например, ВНТК-2025"
    required: true
  grade_level:
    type: select
    label: "Класс участников"
    options:
      - { value: "5-7", label: "5-7 класс" }
      - { value: "8-9", label: "8-9 класс" }
      - { value: "10-11", label: "10-11 класс" }
    default: "10-11"

files:
  works:
    type: folder
    label: "Папка с работами"
    accept: [".pdf", ".docx"]
    max_total_size_mb: 256
    required: true

outputs:
  - id: report
    type: docx
    label: "Сводное заключение"
    filename: "report.docx"
    primary: true
  - id: per_work
    type: zip
    label: "Заключения по каждой работе"
    filename: "per_work.zip"

runtime:
  docker:
    base_image: "python:3.12-slim"
    setup:
      - "pip install -r requirements.txt"
    entrypoint: ["python", "agent.py"]
  limits:
    max_runtime_minutes: 60   # 46 работ × ~60 секунд
    max_memory_mb: 1024
    max_cpu_cores: 2
  # llm: real-версия включит это, stub пока без LLM
```

### science_agent

id: science-agent

  - id: report

  - id: bibtex
    type: any

  docker:

## Stub-семантика

### proverka_stub

`agent.py`:
1. Читает `params["competition"]` и `params["grade_level"]` для лога.
2. Сканирует `agent.input_dir("works")` — каждая подпапка считается отдельной работой.
3. Для каждой подпапки:
   - `agent.progress(i / N, f"Работа {i+1} из {N}: {folder.name}")`
   - Подсчитывает количество файлов и общий размер
   - `agent.item_done(folder.name, summary=f"{n} файлов, {size_kb} КБ", data={...})`
4. Собирает `report.docx` (python-docx) с таблицей: работа / файлов / размер / mock-оценка по чек-листу (4 случайных пункта).
5. Архивирует mock per-work `.docx` (короткая заглушка с именем работы) в `per_work.zip`.
6. `agent.result(artifacts=[{"id": "report", "path": "report.docx"}, {"id": "per_work", "path": "per_work.zip"}])`.

### science_agent_stub

`agent.py`:
1. Читает `params["topic"]`, `params["max_papers"]`, `params["language"]`.
2. Генерирует mock-набор из `min(max_papers, 10)` "статей" (фикс. список на тему `ML/AI`, чтобы каждая работа выглядела правдоподобно).
3. Для каждой статьи:
   - `agent.progress(i / N, f"Анализ {i+1}/{N}: {paper.title}")`
   - `agent.item_done(paper.id, summary=paper.title, data={"score": 0.x, "year": 2024})`
4. `report.docx`: тема + N статей с аннотациями (mock-аннотации повторяют topic).
5. `sources.bib`: BibTeX-блоки для каждой статьи.
6. `agent.result(...)`.
7. `runtime.llm` включён, но stub LLM не зовёт (доказывает что `ephemeral_token` распространяется до контейнера, проверим в логах).

## Публикация

stub-агенты живут в `svpeditor/mirea-agent-portal-proverka-stub` и `svpeditor/mirea-agent-portal-science-stub` — отдельные публичные репозитории (как `mirea-agent-portal-echo-test`). При публикации через `/api/admin/agents` backend клонирует их и собирает образ.

Для local-dev: оба stub зеркалятся в monorepo как `agents/proverka_stub/` и `agents/science_agent_stub/` — для `portal-sdk-run-local` без портала.
