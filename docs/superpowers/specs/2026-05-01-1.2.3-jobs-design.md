# План 1.2.3 «Запуск задач» — jobs + Docker runner + WebSocket (Спек)

**Статус:** дизайн зафиксирован, готов к написанию плана реализации.
**Часть Спека 1.** Четвёртый под-план декомпозиции Плана 1.2:

- 1.2.2 Реестр агентов — `tabs` / `agents` / `agent_versions` + Git-clone + Docker build ✅ (PR #3 на ревью)

## 1. Цель и контекст

После merge 1.2.2 в реестре есть готовые `agent_versions` со собранными Docker-образами (`portal/agent-<slug>:v<sha7>`). Цель 1.2.3 — научить портал **запускать эти образы на пользовательских inputs** и стримить прогресс юзеру в реальном времени по контракту v0.1 (`docs/contract.md`).

После merge 1.2.3:
- Любой залогиненный юзер шлёт `POST /api/agents/{slug}/jobs` с params + input-файлами → получает `job_id`.
- Worker запускает контейнер агента в изоляции (`--network=none`, read-only rootfs, ресурс-лимиты из manifest).
- Юзер подключается к `WS /api/jobs/{id}/stream`, видит события `started/progress/item_done/result/failed/error/log` ровно как они приходят от агента.
- На `result` юзер скачивает артефакты через `GET /api/jobs/{id}/outputs/{file_id}`.

**Что становится возможным после merge 1.2.3:**
- Параллельно стартует 1.2.4 (OpenRouter-прокси) — добавит сетевой доступ агента к LLM через прокси портала с квотами. Сейчас агенты работают только с теми ресурсами, что в их образе.
- Frontend (1.3) получает рабочий API: страница «загрузить файлы → нажать запустить → видеть прогресс».
- Перенос реального агента (`proverka` — План 1.4) становится осмысленным.

## 2. Scope

### Входит ✅

- Таблицы `jobs`, `job_events`, `job_files` + Alembic миграция `0003_jobs`.
- Хранилище входов и выходов: локальный диск через абстракцию `FileStore` (под будущий S3 backend).
- API endpoints (auth required для всех):
  - `POST /api/agents/{slug}/jobs` — multipart: `params` (JSON) + N input файлов → создаёт job, status=`queued`, enqueue в worker.
  - `GET /api/jobs` — пагинация по своим job'ам юзера.
  - `GET /api/jobs/{id}` — job + последние N events (для initial load до WS подписки).
  - `GET /api/jobs/{id}/events?since=<seq>` — REST-фоллбэк на event feed (если WS не доступен).
  - `WS /api/jobs/{id}/stream?since=<seq>` — push events в реальном времени.
  - `GET /api/jobs/{id}/outputs/{file_id}` — скачать артефакт.
  - `POST /api/jobs/{id}/cancel` — отменить (owner или admin).
- Worker `apps/portal-worker/`:
  - Новая RQ-очередь `jobs` (отдельно от `builds`).
  - Функция `run_job(job_id)`: атомарный лок `queued→running`, материализация input-файлов на host, `docker run` с mounts, парсинг JSONL stdout, persist events, publish в Redis pub/sub, верификация output файлов, финализация status.
- WebSocket bridge:
  - На connect: подгрузить events `seq > since` из БД (resync), затем подписаться на Redis pub/sub `job:{id}` и форвардить.
  - Heartbeat ping каждые 30с.
  - Auth через cookie (тот же middleware что REST), проверка job ownership при handshake.
- Безопасность:
  - `--network=none` (агент не имеет интернета — LLM пойдёт через 1.2.4).
  - `--read-only` rootfs + `--tmpfs /tmp`.
  - `--memory`/`--cpus` из `manifest.runtime.limits`.
  - Timeout: SIGTERM после `max_runtime_minutes`, SIGKILL через 10с.
  - File size lim: input total ≤ из manifest или `MAX_JOB_INPUT_BYTES` env (default 100 MB).
- pytest-тесты: testcontainers Postgres+Redis + реальный Docker E2E на echo-агенте, ≥85% coverage.
- README в `apps/portal-api/` дополнен секцией «Запуск задачи» — curl + `wscat` сценарий от создания job до скачивания артефакта.
- docker-compose обновлён: тот же worker слушает обе очереди (`builds` + `jobs`).

### Сознательно отложено ⏭

- LLM proxy + budget enforcement → 1.2.4.
- Frontend → 1.3.
- Cron / scheduled jobs / повторные запуски по расписанию → отдельный план позже.
- Auto-retry на `failed` (только manual через `POST /api/jobs/{id}/retry`, не в этом плане).
- Job priority / queue weights — все job'ы FIFO в одной очереди.
- Multi-tenant квоты на запуск (max running jobs per user) — добавим когда появится реальная нагрузка.
- Worker autoscaling (несколько worker процессов) — single worker per host пока хватает.
- Streaming больших output файлов из контейнера через side-car volume — все файлы пишутся в bind-mount, читаются ПОСЛЕ exit.

### Сознательно НЕ делаем 🚫

- Audit log в отдельной таблице (`structlog` + `jobs.created_by_user_id` хватает).
- Soft-delete для job/files.
- Шифрование артефактов at-rest.
- WebSocket-rooms для broadcast нескольким наблюдателям (один WS = один владелец-job).
- Server-Sent Events альтернатива WebSocket — WebSocket покрывает оба направления (cancel в будущем).
- Pre-validation params против `manifest.inputs` JSON-schema на стороне API. Делаем минимум: проверяем что params валидный JSON. Полная валидация — внутри агента (он лучше знает свою схему).

## 4. Модель данных

### Таблица `jobs`

```sql
CREATE TABLE jobs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_version_id UUID NOT NULL REFERENCES agent_versions(id) ON DELETE RESTRICT,
    created_by_user_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    status TEXT NOT NULL CHECK (status IN ('queued','running','ready','failed','cancelled')),
    params_jsonb JSONB NOT NULL DEFAULT '{}'::jsonb,
    started_at TIMESTAMPTZ,
    finished_at TIMESTAMPTZ,
    exit_code INTEGER,
    error_msg TEXT,
    error_code TEXT,  -- 'agent_failed' | 'timeout' | 'output_missing' | 'docker_error' | 'cancelled'
    output_summary_jsonb JSONB,  -- содержимое последнего event 'result' (artifacts list)
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX jobs_user_created_idx ON jobs(created_by_user_id, created_at DESC);
CREATE INDEX jobs_status_idx ON jobs(status) WHERE status IN ('queued','running');
CREATE INDEX jobs_agent_version_idx ON jobs(agent_version_id);
```

### Таблица `job_events`

Append-only, для replay по WS-reconnect и REST-фоллбэка.

```sql
CREATE TABLE job_events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    seq INTEGER NOT NULL,  -- 1, 2, 3... per job, ASSIGNED BY WORKER
    ts TIMESTAMPTZ NOT NULL DEFAULT now(),
    event_type TEXT NOT NULL,  -- 'started','progress','item_done','result','failed','error','log'
    payload_jsonb JSONB NOT NULL,
    UNIQUE (job_id, seq)
);

CREATE INDEX job_events_job_seq_idx ON job_events(job_id, seq);
```

### Таблица `job_files`

```sql
CREATE TABLE job_files (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    kind TEXT NOT NULL CHECK (kind IN ('input','output')),
    filename TEXT NOT NULL,  -- имя как видит агент: input/<filename> или output/<filename>
    content_type TEXT,
    size_bytes BIGINT NOT NULL,
    sha256 TEXT NOT NULL,
    storage_key TEXT NOT NULL,  -- путь в FileStore (s3-key или local relative path)
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (job_id, kind, filename)
);

CREATE INDEX job_files_job_kind_idx ON job_files(job_id, kind);
```

**Замечание про `seq`:** worker инкрементирует локальный счётчик при каждой записи event. Атомарность — `INSERT ... ON CONFLICT (job_id, seq) DO NOTHING` не нужна, потому что только один worker запускает job (атомарный лок status в шаге 1).

## 5. FileStore

Абстракция:

```python
class FileStore(Protocol):
    async def put(self, key: str, data: AsyncIterable[bytes]) -> tuple[int, str]:
        """Returns (size_bytes, sha256_hex)."""
    async def get(self, key: str) -> AsyncIterable[bytes]: ...
    async def open_path(self, key: str) -> Path:
        """For Docker bind-mount: возвращает абсолютный путь на host."""
    async def delete(self, key: str) -> None: ...
```

**Local backend:**
- Корень: `/var/portal-files/` (volume в docker-compose).
- Layout: `{job_id}/{input|output}/{filename}` — человекочитаемо, легко дебажить.
- `open_path` возвращает `/var/portal-files/{job_id}/...` напрямую.

**S3 backend (заготовка, не имплементируем в 1.2.3):**
- При вызове `open_path` — стянет в локальный tmpdir и вернёт. Для input — на старте job'а, для output — на момент `GET /jobs/{id}/outputs/{file_id}`.

`FileStore` инстанс выбирается по `Settings.file_store_backend = "local" | "s3"`, в 1.2.3 только `local`.

## 6. API endpoints

Все требуют `Depends(get_current_user)` (любой залогиненный юзер).

### `POST /api/agents/{slug}/jobs`

Multipart form-data:
- `params: str` (form field) — JSON-строка, валидируется как валидный JSON (содержимое — не валидируется по manifest.inputs schema).
- N полей с input-файлами, имя поля = `inputs[<filename>]` (где `<filename>` — relative path под `$INPUT_DIR/`, как ожидает агент).

Сервис:
1. Найти агента по slug, проверить `enabled=True AND current_version_id IS NOT NULL` (иначе 404).
2. Загрузить version, проверить `status='ready'` (иначе 409 `agent_not_ready`).
3. Проверить лимит: суммарный размер inputs ≤ `Settings.max_job_input_bytes` (иначе 413 `inputs_too_large`).
4. Транзакция:
   - INSERT в `jobs` со status=`queued`, `agent_version_id=version.id`, `params_jsonb=parsed_params`.
   - Для каждого input файла: stream в FileStore → INSERT в `job_files` с `kind='input'`, `sha256`, `size_bytes`, `storage_key`.
   - Commit.
5. Enqueue в RQ: `jobs_queue.enqueue("portal_worker.tasks.run_job.run_job", str(job.id), job_timeout=settings.job_timeout_seconds + 60)`.

Ответ 202:
```json
{"job": {"id": "...", "status": "queued", "agent_slug": "echo"}}
```

Ошибки:
- 404 `agent_not_found` — агента нет, не enabled, или нет current_version.
- 409 `agent_not_ready` — current_version не в `status='ready'`.
- 413 `inputs_too_large`.
- 400 `params_invalid_json`, `input_filename_invalid` (попытка `../`-пути).

### `GET /api/jobs?limit=20&before=<job_id>`

Пагинация по своим job'ам юзера, отсортированным `created_at DESC`. Cursor-based: `before=` — id последнего из предыдущей страницы (исключительно). Возврат — `JobListItemOut[]` (без events, без output_summary).

### `GET /api/jobs/{id}`

Job-owner или admin. Возвращает полный `JobDetailOut`:
- Поля `jobs.*`
- `agent: {slug, name}` (join'им)
- `events_count: int`
- `last_event_seq: int | null`

### `GET /api/jobs/{id}/events?since=<seq>&limit=200`

Job-owner или admin. Возвращает массив events с `seq > since`. Без долгого polling — это чистый снапшот. Для real-time юзер должен подписаться на WS.

### `WS /api/jobs/{id}/stream?since=<seq>`

Job-owner или admin (auth-check на handshake — кука как в REST, иначе close 4401).

Поток:
1. На connect: query все events `seq > since` из БД и отправить как `[{seq, ts, type, payload}, ...]` (один JSON-кадр).
2. SUBSCRIBE на Redis канал `job:{id}`. Каждое сообщение — один JSON event, форвардим как WS text frame.
3. Если job уже в финальном статусе (`ready|failed|cancelled`) — после resync close с code 1000, без подписки.
4. Heartbeat: app-level `{"type":"ping"}` JSON-кадр каждые 30с (FastAPI WebSocket не имеет встроенного protocol-ping helper; клиент шлёт `{"type":"pong"}` в ответ; если pong не пришёл за 10с — close 1011).

Завершение: на финальном event'е (`result`/`failed`) — сервер close с code 1000. На `POST /jobs/{id}/cancel` снаружи — push event `cancelled` в pub/sub, потом close 1000.

### `GET /api/jobs/{id}/outputs/{file_id}`

Job-owner или admin. Скачать output-файл (`Content-Disposition: attachment; filename="<filename>"`). 404 если file_id не принадлежит job или kind!='output'.

### `POST /api/jobs/{id}/cancel`

Job-owner или admin. Если job в `queued` → атомарно перевести в `cancelled`, удалить из RQ. Если в `running` → пометить флаг в Redis (`job:{id}:cancel`), worker увидит и сделает `docker stop --time=10`. Если уже в финальном — 409 `already_finished`.

Idempotent: повторный POST на cancelled → 200 noop.

## 7. Worker pipeline

`apps/portal-worker/portal_worker/tasks/run_job.py`:

```python
def run_job(job_id: str) -> None:
    """RQ task: атомарный лок → docker run → JSONL parse → finalize."""
    settings = get_settings()
    engine = make_engine(settings)
    session_factory = make_session_factory(engine)
    file_store = make_file_store(settings)
    redis = Redis.from_url(str(settings.redis_url))
    log = structlog.get_logger().bind(job_id=job_id)
    vid = uuid.UUID(job_id)
    workdir = Path(f"/tmp/portal-job-{vid}")

    try:
        # 1. Atomic lock queued → running + load joined data
        with session_factory() as session:
            row = session.execute(text("""
                UPDATE jobs j
                SET status='running', started_at=:now
                FROM agent_versions av, agents a
                WHERE j.id=:vid AND j.status='queued'
                  AND av.id=j.agent_version_id AND a.id=av.agent_id
                RETURNING j.params_jsonb, av.docker_image_tag, av.manifest_jsonb,
                          a.slug AS agent_slug
            """), {"vid": vid, "now": datetime.now(UTC)}).first()
            session.commit()
            if row is None:
                log.info("job_not_queued_or_missing")
                return
            manifest = Manifest.model_validate(row.manifest_jsonb)

        # 2. Materialize inputs to host
        input_dir = workdir / "input"
        output_dir = workdir / "output"
        input_dir.mkdir(parents=True)
        output_dir.mkdir(parents=True)
        params_path = workdir / "params.json"
        params_path.write_text(json.dumps(row.params_jsonb))
        with session_factory() as session:
            for jf in session.execute(text(
                "SELECT filename, storage_key FROM job_files "
                "WHERE job_id=:vid AND kind='input'"
            ), {"vid": vid}).all():
                _materialize(file_store, jf.storage_key, input_dir / jf.filename)

        # 3. Docker run with stream
        seq = 0
        for event in _docker_run_and_stream(
            image_tag=row.docker_image_tag,
            input_dir=input_dir, output_dir=output_dir, params_path=params_path,
            limits=manifest.runtime.limits,
            timeout_seconds=manifest.runtime.limits.max_runtime_minutes * 60,
            cancel_check=lambda: redis.exists(f"job:{vid}:cancel"),
        ):
            seq += 1
            with session_factory() as session:
                session.execute(text("""
                    INSERT INTO job_events (job_id, seq, ts, event_type, payload_jsonb)
                    VALUES (:vid, :seq, :ts, :type, :payload)
                """), {
                    "vid": vid, "seq": seq,
                    "ts": datetime.now(UTC),
                    "type": event["type"],
                    "payload": json.dumps(event),
                })
                session.commit()
            redis.publish(f"job:{vid}", json.dumps({"seq": seq, "event": event}))

        # 4. Finalize (called from inside _docker_run_and_stream when last event seen)
        # Actual finalize logic — verify outputs exist + persist + cleanup
        # ...
    except CancelledError:
        _finalize_cancelled(session_factory, redis, vid)
    except DockerRunError as exc:
        _finalize_failed(session_factory, redis, vid, code=exc.code, msg=exc.msg)
    except Exception as exc:
        log.error("run_job_unexpected", exc_info=True)
        _finalize_failed(session_factory, redis, vid, code="docker_error", msg=str(exc))
    finally:
        shutil.rmtree(workdir, ignore_errors=True)
        engine.dispose()
        redis.delete(f"job:{vid}:cancel")
```

Stdout стрим читаем через `container.logs(stream=True, follow=True)`. Каждая строка — попытка `json.loads`. Невалидный JSON → emit `error` event и продолжить. Каждые 2с проверять `cancel_check()` и `container.reload()` для `status`.

### Output verification

После container exit (или final event):
- Для каждого `manifest.outputs[*].filename` — проверить что файл существует в `output_dir`. Иначе → `error_code='output_missing'`, `failed`.
- Для каждого фактически найденного файла в `output_dir` (включая что не объявлены в manifest):
  - INSERT в `job_files` с `kind='output'`, sha256, storage_key.
  - Скопировать через FileStore.put().

`output_summary_jsonb` = payload последнего `result` event'а (если был).

## 8. WebSocket bridge

В `portal_api/routers/jobs_ws.py`:

```python
@router.websocket("/jobs/{job_id}/stream")
async def stream_job(
    websocket: WebSocket,
    job_id: uuid.UUID,
    since: int = 0,
    db: AsyncSession = Depends(get_db),
    user: User = Depends(get_current_user_ws),
):
    # 1. Проверка job ownership
    job = await job_service.get_for_user(db, job_id, user)
    if job is None:
        await websocket.close(code=4404, reason="job_not_found")
        return
    await websocket.accept()

    # 2. Resync — все events seq > since
    events = await job_event_service.list_since(db, job_id, since)
    await websocket.send_json({"type": "resync", "events": events})

    # 3. Если job уже в финальном — close
    if job.status in ("ready", "failed", "cancelled"):
        await websocket.close(code=1000)
        return

    # 4. Subscribe to Redis pub/sub + heartbeat
    redis = await aioredis.from_url(settings.redis_url)
    pubsub = redis.pubsub()
    await pubsub.subscribe(f"job:{job_id}")

    async def reader():
        async for msg in pubsub.listen():
            if msg["type"] == "message":
                await websocket.send_text(msg["data"])
                # Если event 'result'/'failed' — close
                event = json.loads(msg["data"])
                if event["event"]["type"] in ("result", "failed"):
                    await websocket.close(code=1000)
                    return

    async def heartbeat():
        while True:
            await asyncio.sleep(30)
            try:
                await websocket.send_json({"type": "ping"})
            except WebSocketDisconnect:
                return

    try:
        await asyncio.gather(reader(), heartbeat())
    except WebSocketDisconnect:
        pass
    finally:
        await pubsub.unsubscribe(f"job:{job_id}")
        await pubsub.close()
        await redis.close()
```

`get_current_user_ws` — копия `get_current_user` но для WebSocket (cookie из handshake headers).

## 12. Риски и митигации

| Риск | Митигация |
|---|---|
| Агент пишет 100 GB в output | `--read-only` rootfs + tmpfs limit + `max_job_output_bytes` check ПОСЛЕ exit (если превышен → `failed: output_too_large`, файлы не сохраняются). |
| Агент висит вечно | Hard timeout: SIGTERM → SIGKILL через 10с. |
| Worker умирает в середине → events потеряны? | События пишутся в БД ПЕРЕД publish. После restart `recover_orphaned_jobs` пометит running как failed. История events в БД сохраняется, юзер видит обрыв на последнем event. |
| WebSocket reconnect → дубликаты events | Клиент шлёт `since=<last_seq>`, сервер отдаёт только seq > since. Pub/sub события приходят с тем же `seq` от worker'а — клиент отбрасывает по `seq > last_seq`. |
| Двойной enqueue одного job_id | Атомарный `UPDATE ... WHERE status='queued' RETURNING` (как в build_agent). Второй worker увидит `row is None` и выйдет. |
| FileStore disk fill | Periodic cleanup job на завершённые > N дней (НЕ в 1.2.3 — добавим в эксплуатации). Сейчас просто лимиты на input/output size. |
| Pub/sub message потеряна между write-DB и publish | Клиент при reconnect делает `since=<last_seen>` → подтянет из БД пропущенное. |
| Race на input file: юзер удаляет job до старта worker'а | DELETE jobs cascade → delete job_files → FileStore key осиротевший. Cleanup job (см. выше). |
| Утечка контейнеров при ошибке worker'а | `--rm` на запуске + label `portal-job={vid}` для bulk cleanup в `recover_orphaned_jobs()`. |
